  <action>
Create a migration that modifies existing RLS policies on both students_score and student_skill_progress to enforce the content gate. The migration must:

**1. Add node_id column to students_score:**
```sql
ALTER TABLE students_score ADD COLUMN IF NOT EXISTS node_id TEXT;
CREATE INDEX IF NOT EXISTS idx_students_score_node_id ON students_score(node_id) WHERE node_id IS NOT NULL;
```

**2. Replace the "Consolidated scores access" ALL policy on students_score** with fine-grained per-command policies:

Drop the overly permissive ALL policy:
```sql
DROP POLICY IF EXISTS "Consolidated scores access" ON students_score;
```

Create separate policies:

a) SELECT -- own rows + service_role + connected teachers (preserve teacher access):
```sql
CREATE POLICY "students_score_select"
  ON students_score FOR SELECT
  USING (
    student_id = (SELECT auth.uid())
    OR (SELECT auth.role()) = 'service_role'
    OR (SELECT auth.uid()) IN (
      SELECT teacher_id FROM teacher_student_connections
      WHERE student_id = students_score.student_id
        AND status = 'accepted'
    )
  );
```

b) INSERT with content gate:
```sql
CREATE POLICY "students_score_insert_gate"
  ON students_score FOR INSERT
  WITH CHECK (
    student_id = (SELECT auth.uid())
    AND (
      is_free_node(node_id)
      OR has_active_subscription((SELECT auth.uid()))
    )
  );
```

c) Keep the existing "Students can delete own scores" DELETE policy unchanged (already fine-grained).

**3. Replace student_skill_progress INSERT and UPDATE policies** with gated versions:

Drop old policies:
```sql
DROP POLICY IF EXISTS "student_skill_progress_insert_own" ON student_skill_progress;
DROP POLICY IF EXISTS "student_skill_progress_update_own" ON student_skill_progress;
```

Create gated INSERT:
```sql
CREATE POLICY "student_skill_progress_insert_gate"
  ON student_skill_progress FOR INSERT
  WITH CHECK (
    student_id = (SELECT auth.uid())
    AND (
      is_free_node(node_id)
      OR has_active_subscription((SELECT auth.uid()))
    )
  );
```

Create gated UPDATE:
```sql
CREATE POLICY "student_skill_progress_update_gate"
  ON student_skill_progress FOR UPDATE
  USING (student_id = (SELECT auth.uid()))
  WITH CHECK (
    student_id = (SELECT auth.uid())
    AND (
      is_free_node(node_id)
      OR has_active_subscription((SELECT auth.uid()))
    )
  );
```

Leave the SELECT and DELETE policies on student_skill_progress unchanged.

**CRITICAL:** Also drop the existing "Teachers can view connected students scores" SELECT policy on students_score. Its teacher subquery is consolidated into the new `students_score_select` policy above, avoiding duplicate permissive SELECT policies (the exact anti-pattern previous migrations fixed):
```sql
DROP POLICY IF EXISTS "Teachers can view connected students scores" ON students_score;
```

-- No UPDATE policy: students_score is append-only; all write paths use INSERT only.

**CRITICAL:** is_free_node(NULL) returns true (verified in Plan 01), so non-trail games with NULL node_id are never blocked.

Use `mcp__supabase__apply_migration` to apply.

  <files>src/services/apiScores.js</files>
  <action>
Modify the `updateStudentScore` function in `src/services/apiScores.js` to include `node_id` in the INSERT row.

Currently the INSERT is:
```javascript
.insert([{ student_id: studentId, score: score, game_type: gameType }])
```

Change to:
```javascript
.insert([{ student_id: studentId, score: score, game_type: gameType, node_id: nodeId || null }])
```

The `nodeId` parameter already exists in the function signature (`nodeId = null`). It is currently only used for rate limiting. Now it will also be stored in the row for RLS gate enforcement.

When `nodeId` is null (non-trail games), the column stores NULL, and `is_free_node(NULL)` returns true so the INSERT is never blocked.

When `nodeId` is a free node ID (e.g., 'treble_1_1'), `is_free_node('treble_1_1')` returns true and the INSERT succeeds.

When `nodeId` is a premium node ID (e.g., 'treble_2_1') and the student has no active subscription, `is_free_node('treble_2_1')` returns false and `has_active_subscription(auth.uid())` returns false, so the INSERT is rejected by RLS.

Do NOT change any other logic in the file. The rate limiting, authorization check, and error handling remain unchanged.
