## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage                                                                         |
| ----- | ----- | ----------------------------------------------------------------------------- |
| xs    | 4px   | Icon gaps, combo badge inner padding (`px-2 py-0.5`)                          |
| sm    | 8px   | Tile horizontal inset from lane edge (`left-4 right-4` = 16px, see exception) |
| md    | 16px  | Tile lane horizontal padding (existing `left-4 right-4`), header px           |
| lg    | 24px  | Hit zone bottom offset reference (`bottom-12` = 48px, see exception)          |
| xl    | 32px  | Setup screen gap between title and Play button                                |
| 2xl   | 48px  | Hit zone height — `h-12` (48px); minimum tap target on portrait               |

Exceptions:

- **Hit zone height: 48px minimum, 64px target on ship-vertical.** `h-12` (48px) is the current value and meets WCAG 2.5.5 minimum (44px). For the ship-vertical path, the planner should evaluate bumping to `h-16` (64px) on phone-portrait only (`sm:h-12` or a conditional class) because the taller aspect ratio gives more room and a larger tap target reduces miss-frustration for 8-year-olds. Decision left to planner after feel-test confirms portrait is viable.
- **44px minimum tap target for interactive elements.** WCAG 2.5.5. The existing 48px hit zone satisfies this. No tile is interactive (tiles are visual only); only the hit zone bottom strip is tapped.

## Portrait Viewport Contract (ship-vertical path)

This section is the primary addendum this phase introduces. It defines the visual targets for the feel-test spike and any subsequent polish, pre-locking the contract so the spike is not biased by ad-hoc decisions during play.

### Floor target device

iPhone SE portrait: **375px × 667px** (CSS viewport, no safe-area insets applied).
Secondary target: iPhone 14 portrait: **390px × 844px**.
These match the CONTEXT.md D-03 spike test surface.

### Lane geometry at floor target (375 × 667)

| Element                      | Current value (landscape)             | Portrait target                                                    | Notes                                                                        |
| ---------------------------- | ------------------------------------- | ------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| Header height                | `h-12` (48px)                         | `h-12` (48px) — unchanged                                          | Fits counter + hearts + combo at 375px width                                 |
| Tile lane height (main area) | viewport_height − header − hit_zone   | 667 − 48 − 48 = **571px**                                          | ~85% of viewport; more vertical travel than landscape gives more time-to-hit |
| Tile lane horizontal padding | `left-4 right-4` (16px each side)     | `left-4 right-4` — unchanged                                       | Tiles span 375 − 32 = **343px** wide; sufficient visual surface              |
| Hit zone height              | `h-12` (48px)                         | 48px minimum; planner may promote to `h-16` (64px) after feel-test | Larger target on portrait is lower-risk polish                               |
| Tile minimum height          | 36px (sixteenth, `TILE_HEIGHTS[1]`)   | 36px — unchanged                                                   | 36px at 343px wide is a legible rectangle; no resize needed                  |
| Tile maximum height          | 80px (whole note, `TILE_HEIGHTS[16]`) | 80px — unchanged                                                   | Whole tile spans 12% of lane height at portrait — proportionate              |

### Tile density concern (spike primary failure mode)

A 4-bar 4/4 pattern = 16 beats. At BPM 80, beats arrive every 750ms. With `laneHeightRef` reading the dynamic portrait height (571px), tiles have ~750ms of travel time per beat interval. At the current BPM range, 4-bar dense patterns will show 3–5 tiles simultaneously in the lane. The feel-test (D-03, D-04) must confirm this density reads as "readable arcade rhythm" not "overwhelming". If density fails: the rotate-prompt path wins (D-07 tie-break defaults to rotate-prompt).

### Header at 375px — layout audit

Header contains (left to right): BackButton | exercise counter text | [combo badge] + [hearts row].

At 375px with `px-4` (16px each side) = 343px usable. Hearts row: 3 × `h-5 w-5` (20px) + `gap-1` (4px) = 68px. Combo badge when shown: ~60px. Counter text: ~40px. BackButton: ~32px. Total worst case: 32 + 40 + 60 + 68 + gaps ≈ 230px — fits within 343px without wrapping. No header changes required.

## Interaction Contract

### Tap target

| Surface             | Min size  | Current                                | Portrait change                                  |
| ------------------- | --------- | -------------------------------------- | ------------------------------------------------ |
| Hit zone (tap)      | 44 × 44px | full-width × 48px                      | Full-width × 48px (or 64px if planner promotes)  |
| Back button         | 44 × 44px | icon in `p-2` wrapper (~36px rendered) | No change — BackButton component owns its sizing |
| Play button (setup) | 44 × 44px | `px-8 py-4` = large button             | No change                                        |

### Pointer events

`onPointerDown` is used (not `onClick`) for the hit zone — this is intentional for low-latency tap response on mobile. Do not change to `onClick`. The game wrapper also attaches `onPointerDown` to the outer `div` as a backup; both are intentional.

### Orientation change while in-progress

Phase 34 RotatePromptOverlay handles the overlay display. On the ship-vertical path (rotate-prompt not shown), if the user rotates to landscape mid-game, the game continues to play in landscape (extra screen space = wider tiles; no breakage). No interaction contract changes needed for this edge case.

### Reduced motion

All tile animations and the hit-zone pulse respect `reducedMotion` from `AccessibilityContext`. The on-fire tile glow (`drop-shadow`) is gated on `!reducedMotion`. No changes to this contract.
