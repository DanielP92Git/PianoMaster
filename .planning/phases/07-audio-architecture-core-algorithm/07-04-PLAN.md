**4. Create onNoteEvent callback for useMicNoteInput:**
This callback replaces the inline detect loop's note matching logic (lines ~1780-1797).

```javascript
const handleMicNoteEvent = useCallback((event) => {
  if (event.type !== 'noteOn') return;

  const note = event.pitch;
  setDetectedNote(note);

  // Game-specific logic: check if detected note matches current question
  if (note && progress.currentNote && note === progress.currentNote.note) {
    handleAnswerSelect(note);
  }
}, [progress.currentNote, handleAnswerSelect]);
```

Note: The `waitingForRelease` / `pendingNextNote` logic (lines 1759-1773) from the old detect loop ran inside the rAF loop and checked audio level directly. This game logic must be preserved but adapted. The simplest approach: keep `waitingForRelease` state, and in the onNoteEvent callback, if `waitingForRelease` is true, ignore note events. Then use a separate `useEffect` watching `audioLevel` from useMicNoteInput to detect when level drops below release threshold and proceed to next note.

Actually, looking more carefully at the flow:
- `waitingForRelease` is set to true after a correct answer in Listen mode
- It waits for audio level to drop below RELEASE_THRESHOLD before showing next note
- This prevents the same sustained note from immediately answering the next question

The adapted approach:
- Keep `waitingForRelease` and `pendingNextNote` state
- In `handleMicNoteEvent`, check `waitingForRelease` — if true, skip processing
- Add a `useEffect` that watches `audioLevel` from useMicNoteInput:
  ```javascript
  useEffect(() => {
    if (waitingForRelease && audioLevel * 100 < RELEASE_THRESHOLD) {
      setWaitingForRelease(false);
      setAnswerFeedback({ selectedNote: null, correctNote: null, isCorrect: null });
      updateProgress({ currentNote: pendingNextNote });
      setPendingNextNote(null);
    }
  }, [waitingForRelease, audioLevel, pendingNextNote, updateProgress]);
  ```
