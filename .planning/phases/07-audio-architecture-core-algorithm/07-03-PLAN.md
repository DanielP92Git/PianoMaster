**4. Modify `cleanup` function (CRITICAL — Pitfall 4 from research):**
```javascript
const cleanup = useCallback(() => {
  stopScheduler();

  if (schedulerIdRef.current) {
    clearTimeout(schedulerIdRef.current);
    schedulerIdRef.current = null;
  }

  // Only close AudioContext if we own it (not shared)
  if (isOwnedContextRef.current && audioContextRef.current && audioContextRef.current.state !== 'closed') {
    audioContextRef.current.close();
    audioContextRef.current = null;
  }

  // If shared context: just null out our ref (provider manages lifecycle)
  if (!isOwnedContextRef.current) {
    audioContextRef.current = null;
  }

  setIsPlaying(false);
  setIsInitialized(false);
}, [stopScheduler]);
```

**5. Modify the mount useEffect:**
```javascript

  initializeAudioContext();
  return cleanup;
}, [initializeAudioContext, cleanup]);
```
No change needed — `initializeAudioContext` will use sharedAudioContext if available.

**6. Modify `resumeAudioContext`:**
The existing logic already handles the case where audioContextRef.current exists. The key change is that `initializeAudioContext` should NOT try to create a new context if one is already provided via sharing. The current fallback (`if (!audioContextRef.current) { await initializeAudioContext(); }`) already works correctly since `initializeAudioContext` will use the shared context.

**7. No changes to any other methods** — they all use `audioContextRef.current` internally, which now points to either the shared or owned context transparently.
