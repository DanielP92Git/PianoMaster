```javascript
performance.mark('findPitch-start');
const [pitch, clarity] = detectorRef.current.findPitch(inputBufferRef.current, sampleRate);
performance.mark('findPitch-end');
performance.measure('findPitch', 'findPitch-start', 'findPitch-end');
```

These marks appear as named segments in the Performance panel's "Timings" track when profiling via Chrome DevTools, making it easy to identify findPitch duration vs frame budget.

Gate the marks behind a check so they have zero overhead in production:
```javascript
const __PERF_MARKS = typeof performance !== 'undefined' && typeof performance.mark === 'function';
```
Define this constant at the module level. In the detect loop:
```javascript
if (__PERF_MARKS) performance.mark('findPitch-start');
const [pitch, clarity] = ...;
if (__PERF_MARKS) performance.mark('findPitch-end');
if (__PERF_MARKS) performance.measure('findPitch', 'findPitch-start', 'findPitch-end');
```

Do NOT modify any detection logic, thresholds, or behavior. Only add instrumentation.

## Profiling Procedure

### Setup
1. Connect your Android device via USB cable
2. Enable USB debugging: Settings > Developer Options > USB debugging
3. On your PC, open Chrome and navigate to `chrome://inspect#devices`
4. Check "Discover USB devices" - your device should appear
5. On the Android device, open Chrome and navigate to your dev server URL (e.g., `http://<your-ip>:5174`)
6. In chrome://inspect, click "Inspect" next to the app tab

### Critical: Disable Screencast
- In the DevTools window that opens, click the screencast toggle (monitor icon, top-left) to **disable** it
- Leaving screencast on artificially drops FPS and skews results

### Record a Profiling Session
1. Open the **Performance** tab in DevTools
2. Navigate to a **Sight Reading** game (primary target — most complex: VexFlow + mic + scoring)
3. Configure with **bass clef notes** (A2-E4 range) — these stress the 4096-sample FFT most
4. Click the Record button (circle icon) in Performance tab
5. Play through a full 10-question exercise session
6. Click Stop
7. Examine:
   - **FPS chart** (top): Look for red bars during active mic detection
   - **Main thread flame chart**: Look for `findPitch` and `getAudioData` segments in the Timings track
   - **Summary tab** (bottom): Note % Scripting vs % Rendering vs % Idle
8. Optional: Open **WebAudio** panel (DevTools > More tabs > WebAudio) to check Render Capacity %

### Secondary Profile (optional)
- Repeat with **Notes Recognition** game to isolate audio cost without VexFlow overhead

### Decision Criteria
- **PASS (no AudioWorklet needed):** FPS stays >= 57fps during active detection, `findPitch` segments are < 3ms each, Scripting < 5% of frame budget
- **FAIL (build AudioWorklet):** Red bars in FPS chart correlating with audio callbacks, or `findPitch` consistently > 5ms, or Scripting > 5% of frame budget

### Report Results
After profiling, report:
1. Device model and Chrome version
2. FPS range observed during active mic detection
3. findPitch average duration (from Timings track)
4. Overall Scripting % from Summary
5. Your assessment: **PASS** or **FAIL**
6. Any other notable findings (VexFlow hotspots, etc.)

If you don't have a mid-range Android device available, you can use Chrome DevTools CPU throttle (4x slowdown) on desktop as a rough approximation — note this in results.
