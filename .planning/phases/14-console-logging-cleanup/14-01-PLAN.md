**Severity rationale:** Using `"warn"` (not `"error"`) per research recommendation. This ensures:
- Developers see the warning in their terminal
- CI does not hard-fail on gated DEV logs that use `// eslint-disable-next-line no-console`
- Husky lint-staged runs ESLint but does not block commits on warnings (only errors block commits)

- Inside functions gated by `import.meta.env.DEV`-sourced boolean flags (like `RHYTHM_DEBUG`, `DEBUG`, `METRONOME_TIMING_DEBUG`) — these will still trigger the lint warning since ESLint cannot evaluate runtime guards
