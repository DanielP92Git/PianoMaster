## Bug 1: Double Cursor

**Root cause:** Two cursor divs render during playback:
1. Inside `RhythmStaffDisplay.jsx` lines 184-202: renders when `showCursor={true}`, uses `cursorProgress` prop (always 0 from parent)
2. Inside `RhythmReadingGame.jsx` lines 675-691: the REAL cursor, positioned via RAF loop + cursorDivRef

The parent passes `showCursor={isPlaying}` (line 671) and `cursorProgress={0}` (line 669). During playback, `isPlaying=true` so RhythmStaffDisplay shows its static cursor at 0%. Meanwhile the parent's cursor div animates correctly.

**Fix:** Pass `showCursor={false}` to RhythmStaffDisplay since the parent manages its own cursor. The parent cursor (lines 675-691) is the correct one — it's positioned via RAF direct DOM manipulation.

## Bug 2: Cursor/Metronome Desync at Beat 4

**Root cause:** The cursor moves linearly across time:
```javascript
const progress = Math.min(elapsed / total, 1);
cursorDivRef.current.style.left = `${progress * 100}%`;
```
(RhythmReadingGame.jsx lines 387-391)

But VexFlow's Formatter spaces notes non-linearly — notes are positioned based on their musical duration, not evenly across the staff width. Additionally, the cursor spans the full parent container width (0% to 100%) while notes occupy only a portion (after clef/time-sig, before the end bar).

The cursor reaches the visual position of beat 4 before the metronome ticks beat 4 because:
- The cursor at 75% of time = 75% of container width
- But beat 4's note X position might be at 65% of container width (VexFlow compressed it)

**Fix approach:** After VexFlow renders, extract the X positions of each note's SVG element. Build a position-to-time mapping. In the RAF loop, interpolate cursor position between note X positions based on elapsed time relative to scheduled beat times. This ensures cursor reaches each note exactly when its beat fires.

Implementation:
- In RhythmStaffDisplay, expose a callback or ref that provides note X positions after render
- In RhythmReadingGame's startPlaying(), use these X positions to map `progress` to actual pixel position instead of linear percentage
- Fallback: if note positions aren't available, keep current linear behavior

**Simplified approach (preferred):** The stave has a known start-X (after clef/time sig) and end-X (before the barline). Map cursor from stave-start-X to stave-end-X over the measure duration, instead of 0% to 100% of the container. This alone fixes most of the desync because VexFlow's Formatter distributes notes proportionally to duration within the stave bounds.

To get stave bounds:
```javascript
// After VexFlow render, query the SVG for stave line bounds
const staveLines = containerRef.current?.querySelectorAll('.vf-stave .vf-line');
// Or use Stave.getNoteStartX() and Stave.getNoteEndX() from VexFlow API
```

Expose staveStartX and staveEndX from RhythmStaffDisplay via a callback prop like `onStaveBoundsReady({ startX, endX, totalWidth })`.

Then in the RAF loop:
```javascript
const containerWidth = cursorDivRef.current.parentElement.offsetWidth;
const startPct = staveStartX / containerWidth;
const endPct = staveEndX / containerWidth;
const cursorPct = startPct + progress * (endPct - startPct);
cursorDivRef.current.style.left = `${cursorPct * 100}%`;
```

## Bug 3: BackButton Stuck Spinning

**Root cause:** `BackButton.jsx` line 15 sets `isNavigating=true` but never resets it. `navigate()` is async — the component stays mounted with the spinner until React Router completes the transition, but if navigation fails or is slow, it's stuck forever.

**Fix:** Remove the `isNavigating` state entirely — it adds complexity without value. The `setTimeout(() => navigate(...), 0)` wrapper is also unnecessary. Simplify to just call `navigate(to)` directly in `handleClick`. If a loading indicator is truly needed, add a 3-second safety timeout that resets `isNavigating` to false.

Simpler approach: Just remove the loading state. Navigation is near-instant in an SPA.
