# Quick Task 260614-5wj: Unify "Listen & Tap" (MetronomeTrainer) HUD

## Problem

Phase 36 ("Game Screen UI Unification") extracted a shared game HUD
(`ProgressBar`, `ScorePill`, etc.) and adopted it across games. The "Listen & Tap"
game — `src/components/games/rhythm-games/MetronomeTrainer.jsx` — was given the
shared HUD **only in trail mode** (decision D-02), so it still renders its own
progress/feedback UI everywhere else. Compared to its sibling tap game
`RhythmReadingGame` (the consistency target), it diverges in three ways:

1. **Header** — big centered `<h1>"Listen & Tap"` + `timeSig • BPM • difficulty`
   subtitle, instead of the slim `BackButton + ProgressBar + ScorePill` header the
   other games use (`RhythmReadingGame.jsx:889`).
2. **Progress + score gated to trail mode** — in free-practice the `ProgressBar`
   and `ScorePill` are hidden and replaced by a custom text counter
   (`MetronomeTrainer.jsx:1417,1429`). Other games show them unconditionally.
3. **Unique live 4-stat footer** — patterns / XP / maxCombo / goodTaps readout
   during play (`MetronomeTrainer.jsx:1475`), which no other game shows. (Those
   stats already surface on the end-of-session `VictoryScreen`.)

## Decision (owner gave discretion)

Fix the genuine UI divergences; keep what is mechanic-appropriate.

- **DO** unify the header to the slim shared pattern, with `ProgressBar` + `ScorePill`
  shown **always** (remove the `nodeId` gating, the text-counter fallback, and the
  title/subtitle).
- **DO** remove the bespoke live 4-stat footer (stats remain on `VictoryScreen`).
- **KEEP** the in-tap-button accuracy feedback (`TapArea`) — it is coupled to the
  metronome's tap mechanic; `FloatingFeedback` is designed for staff games and would
  be a downgrade here.
- **KEEP** the phase guidance text (instructional, not a styling divergence).
- No scoring/mechanics/state changes — presentation only.
