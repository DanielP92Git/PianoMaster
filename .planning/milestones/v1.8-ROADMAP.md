# Milestone v1.8: App Monetization

**Status:** IN PROGRESS (Phases 11-12 complete)
**Phases:** 11-16
**Total Plans:** 6 (across phases 11-12 complete, 13-16 planned)

## Overview

Add freemium monetization so parents can subscribe to unlock the full trail, with COPPA-safe purchase flows and dual-market support (Israel + USA).

## Phases

### Phase 11: Legal, Gate Design, and Processor Setup

**Goal**: All pre-code decisions are locked — payment processor is selected and verified for Israel support, the free tier boundary is validated against actual unit files, the parental consent email names the payment processor, and no child PII flows to the billing system
**Depends on**: Phase 10 (v1.7 complete) or parallel with v1.7 wrap-up
**Requirements**: PAY-01, COMP-03, COMP-04
**Status**: Complete
**Success Criteria** (what must be TRUE):
  1. Payment processor account is registered with sandbox active and payout path to an Israeli bank confirmed — this is documented before any billing code is written
  2. The parental consent email shown to parents names the payment processor explicitly — a parent receiving the email today knows which company handles their payment data
  3. No child personal data (name, age, grade) is included in any payment processor API call — only the parent's email address is transmitted for billing
  4. The free tier boundary (Unit 1 per path) is verified against `trebleUnit1.js`, `bassUnit1.js`, `rhythmUnit1.js` node counts — the exact count of free nodes is documented
**Plans**: 3 plans
- [x] 11-01-PLAN.md -- Subscription config and consent modal
- [x] 11-02-PLAN.md -- Processor placeholder (TBD documented)
- [x] 11-03-PLAN.md -- Lemon Squeezy processor confirmation

### Phase 12: Database Schema and RLS

**Goal**: The subscription database foundation exists — tables are created with correct RLS so the client can only read subscription state, all writes are restricted to the webhook service role, the pricing data is seeded, and the content gate is enforced at the database layer not only in React
**Depends on**: Phase 11
**Requirements**: SUB-01, SUB-02, SUB-03, SUB-04, GATE-03
**Status**: Complete
**Success Criteria** (what must be TRUE):
  1. A logged-in student calling the Supabase client directly cannot INSERT or UPDATE rows in `parent_subscriptions` — only the webhook service role key can write subscription state
  2. The `subscription_plans` table contains monthly and yearly rows for both ILS and USD — a fresh database query returns all four price rows without seed data being re-run
  3. `src/config/subscriptionConfig.js` exists with an `isFreeNode(node)` function — changing the free tier boundary requires editing exactly one file in one place
  4. A student who bypasses React's `isPremium` check and attempts to save a score on a premium node is blocked at the database level — the INSERT is rejected by RLS
**Plans**: 2 plans
- [x] 12-01-PLAN.md -- Database schema and RLS
- [x] 12-02-PLAN.md -- Content gate RLS

### Phase 13: Payment Webhook and Service Worker

**Goal**: Subscription lifecycle events from the payment processor are received, verified, and applied to the database — the webhook is idempotent and handles duplicate delivery, the service worker never caches subscription state, and the cache version is bumped for the monetization deploy
**Depends on**: Phase 12
**Requirements**: PAY-02, PAY-03, PAY-04, COMP-01, COMP-02
**Status**: Not started
**Success Criteria** (what must be TRUE):
  1. Sending a `subscription.created` test event via the payment processor's sandbox sets `status = 'active'` in `parent_subscriptions` — the subscription activates without manual intervention
  2. Sending the same event twice produces one row in `parent_subscriptions`, not two — the webhook handler is idempotent
  3. A webhook request with an invalid or missing signature header returns a 400 error and no database write occurs — forged webhooks are rejected
  4. Subscription status API responses are never served from the service worker cache — a hard-refreshed PWA always fetches current subscription state from the network
**Plans**: TBD

### Phase 14: Subscription Context and Service Layer

**Goal**: React components can read subscription status globally — `SubscriptionContext` provides `isPremium` with staleTime: 0, a Supabase Realtime channel invalidates the query the moment the webhook writes an update, and all components share one source of truth
**Depends on**: Phase 13
**Requirements**: SVC-01, SVC-02, SVC-03
**Status**: Not started
**Success Criteria** (what must be TRUE):
  1. Any component in the app tree can call `useSubscription()` and receive the current `isPremium` boolean — no prop drilling, no duplicate fetch calls
  2. After a parent completes checkout and the webhook delivers, the child's trail view unlocks within seconds without the child refreshing the page — the Realtime channel triggers cache invalidation automatically
  3. Opening the app after a period offline and then reconnecting re-fetches subscription status immediately — the `staleTime: 0` setting prevents a stale "not premium" state from persisting after a payment
**Plans**: TBD

### Phase 15: Trail Content Gating UI

**Goal**: Students see a clearly gated trail where subscription-locked nodes look different from prerequisite-locked nodes, and tapping a locked premium node shows a child-appropriate message with no pricing or purchase UI visible
**Depends on**: Phase 14
**Requirements**: GATE-01, GATE-02, CHILD-01, CHILD-02
**Status**: Not started
**Success Criteria** (what must be TRUE):
  1. A student on a free account sees nodes beyond Unit 1 displayed with a distinct gold lock overlay — the locked appearance is visually different from the gray prerequisite lock on nodes they haven't unlocked yet
  2. Tapping a subscription-locked node as a student opens a modal that contains no prices, no "buy" buttons, and no payment form — the child sees only a friendly message and a "Tell a grown-up" type call-to-action
  3. No navigation path reachable by a logged-in student leads to a pricing page, checkout form, or payment interface — the boundary between child session and parent purchase flow is enforced at every route
  4. A student with an active parent subscription sees all trail nodes as available (subject to prerequisite unlock) — the gold lock does not appear for premium subscribers
**Plans**: TBD

### Phase 16: Parent-Facing Pages and Checkout

**Goal**: A parent can view pricing in their local currency, initiate checkout via the selected payment processor without leaving the PWA, confirm their subscription activated, and cancel within the app — completing the full purchase lifecycle
**Depends on**: Phase 15
**Requirements**: PARENT-01, PARENT-02, PARENT-03, PARENT-04, PARENT-05
**Status**: Not started
**Success Criteria** (what must be TRUE):
  1. A parent visiting `/subscribe` sees prices displayed in ILS if their account is Israel-associated or USD otherwise — they do not need to manually switch currency
  2. Clicking a plan on the pricing page opens the payment processor's checkout — the parent completes payment without navigating away from the PWA
  3. After completing checkout, the parent lands on a success page that confirms their subscription is active — the page polls until the webhook confirms activation (up to 10 seconds) before showing the confirmation
  4. A parent can cancel their subscription from within the app at `/parent-portal` — cancellation completes without the parent leaving the PWA to visit an external billing portal
  5. After cancellation, the child retains full premium access until the end of the current billing period — access does not revoke immediately on the day of cancellation
