## Component Inventory

This phase adds one new step component to `UnifiedGameSettings` and passes a new prop to `VexFlowStaffDisplay`. No new page-level components.

### New: `KeySignatureSelection` step component

**Location:** `src/components/games/sight-reading-game/components/KeySignatureSelection.jsx`

**Renders inside:** `UnifiedGameSettings` wizard, as step 2 (after ClefSelection, before NoteSelection).

**Step config entry in `PreGameSetup.jsx`:**
```js
{
  id: 'keySignature',
  title: 'gameSettings.steps.labels.keySignature',
  component: 'KeySignatureSelection',
}
```

**Layout pattern:** Grid of selectable option cards Рђћ same pattern as `ClefSelection`. 7 options in a 2-column grid (or wrapping flex on mobile). Each card:
- Container: `rounded-xl border border-white/20 bg-white/10 p-3 text-center cursor-pointer transition-colors hover:bg-white/15 min-h-[44px]`
- Selected state: `border-indigo-400 bg-indigo-600/30`
- Key name: `text-sm font-semibold text-white`
- Accidental count badge: `text-xs text-indigo-300` (e.g. "1 sharp", "2 flats", "no accidentals")

**Options (7 total, in display order):**

| Display Label | Badge | `keySignature` value |
|---------------|-------|---------------------|
| C major | no accidentals | `null` (default) |
| G major | 1 sharp | `'G'` |
| D major | 2 sharps | `'D'` |
| A major | 3 sharps | `'A'` |
| F major | 1 flat | `'F'` |
| BРЎГ major | 2 flats | `'Bb'` |
| EРЎГ major | 3 flats | `'Eb'` |

**Default:** C major (`null`) pre-selected. Matches "no key signature" = current behavior.

**Interaction:** Single-select. Selecting a key immediately calls `updateSetting('keySignature', value)`. No confirmation required. Step is always valid (all 7 options are valid choices).

**Auto-filter note pool:** When a non-C key is selected, the NoteSelection step (next step) will pre-filter `selectedNotes` to in-key pitches. The auto-filter happens in `PreGameSetup.jsx` within the `onUpdateSettings` handler for `keySignature`, not inside `KeySignatureSelection` itself. The user sees the effect when they advance to the Notes step.

### Modified: `VexFlowStaffDisplay` Рђћ key signature glyph area

**No new UI chrome.** The key signature glyph (e.g., one sharp on the F line for G major) is rendered by VexFlow's `stave.addKeySignature()` as an SVG element. No CSS or Tailwind required. Standard VexFlow rendering Рђћ no custom styling, no size override.

The rendered glyph is black on the white staff SVG, consistent with all other notation elements. No color theming applied to notation SVG content.

### Key Signature Step (new)

| State | Visual |
|-------|--------|
| Default (C major pre-selected) | C major card: `bg-indigo-600/30 border-indigo-400`. All others: `bg-white/10 border-white/20` |

### VexFlow Staff Display Рђћ key signature rendering states

| Condition | Visual |
|-----------|--------|
| `keySignature` is null or 'C' | Staff renders with no key signature glyphs Рђћ identical to current behavior |
| `keySignature` is 'G' | One sharp glyph on F line appears between clef and time signature on bar 0 |
| `keySignature` is 'D' | Two sharp glyphs on F and C lines on bar 0 |
| `keySignature` is 'A' | Three sharp glyphs on F, C, G lines on bar 0 |
| `keySignature` is 'F' | One flat glyph on B line on bar 0 |
| `keySignature` is 'Bb' | Two flat glyphs on B and E lines on bar 0 |
| `keySignature` is 'Eb' | Three flat glyphs on B, E, A lines on bar 0 |
| Note covered by active key sig | Note renders without accidental symbol (suppressed by `applyAccidentals`) |
| Note deviating from key sig (natural) | Natural sign `РЎ«` appears on note head |
| Multi-bar pattern, bar 1+ | No key signature glyph (first bar only Рђћ standard convention) |
| Grand staff (`clef='both'`) | Key signature glyph appears on both treble and bass staves, bar 0 only |

## Accessibility Contract

All requirements from `AccessibilityContext` must hold for the new step component:

- **High contrast mode:** Key option cards use `border-2 border-white` instead of `border-white/20` when `highContrast` is true.
- **Reduced motion:** No animation on key option selection (selection is instant state change only). The step transition animation in `UnifiedGameSettings` already respects `useMotionTokens()` Рђћ new step inherits this.
- **Screen reader:** Each key option card must be a `<button>` with `aria-pressed` reflecting selected state. Badge text (e.g., "1 sharp") must be inside the button so it is read by screen readers.
- **RTL (Hebrew):** Key option grid uses `flex-wrap` or CSS grid Рђћ both are direction-agnostic. No absolute positioning used. Text labels will mirror naturally in RTL layout.
- **Touch targets:** All key option buttons: `min-h-[44px] min-w-[44px]` enforced.

## RTL / i18n Notes

- Hebrew key name translations should use standard music theory terms in Hebrew (e.g., "ОЊОЋ ОъОќ'ОЋОе" for C major, "ОАОЋОю ОъОќ'ОЋОе" for G major).
- Accidental badge translations: "ОЊОЎОљОќ" (sharp), "ОЉОъОЋОю" (flat), count prefix before noun (RTL-safe with Unicode bidirectional).
- The flat symbol РЎГ and sharp symbol РЎ» are neutral Unicode characters Рђћ display correctly in both LTR and RTL.
