  <action>
    Make two targeted modifications to `public/sw.js`:

    **Modification 1: Bump cache version (line 4)**

    Change:
    ```javascript
    const CACHE_NAME = "pianomaster-v5";
    ```
    To:
    ```javascript
    const CACHE_NAME = "pianomaster-v6";
    ```

    This is the ONLY change needed for cache busting — the existing `activate` handler already deletes caches not in `CACHE_WHITELIST`, and `CACHE_WHITELIST` references `CACHE_NAME`. Bumping from v5 to v6 causes old v5 caches to be purged on the next service worker activation.

    **Modification 2: Add REST API exclusion function**

    Add a new function `isRestApiEndpoint(url)` immediately after the existing `isAuthEndpoint(url)` function (after line ~56). Follow the same pattern as `isAuthEndpoint`:

    ```javascript
    /**
     * Check if a URL is a Supabase REST API endpoint that should not be cached.
     * REST API responses (including subscription status) must always be fetched
     * from the network to prevent stale data.
     * @param {URL} url - The URL object to check
     * @returns {boolean} - True if the URL is a REST API endpoint
     */
    function isRestApiEndpoint(url) {
      if (!url.hostname.includes('supabase.co')) {
        return false;
      }
      return url.pathname.startsWith('/rest/');
    }
    ```

    **Modification 3: Update shouldCache condition**

    In the fetch event handler's network-first section (around line 225), find:
    ```javascript
    const shouldCache = matchesPattern && !isAuth;
    ```

    Change to:
    ```javascript
    const shouldCache = matchesPattern && !isAuth && !isRestApiEndpoint(url);
    ```

    **Modification 4: Add REST API exclusion to the offline fallback path**

    In the `catch` block of the same fetch handler (around line 244), after the existing auth endpoint check, add a similar check for REST API endpoints. When offline and the request is for a REST API endpoint, return a generic offline JSON response (do NOT serve from cache). This ensures subscription status is never served stale even during offline fallback.

    After the existing block:
    ```javascript
    // SECURITY: Never serve auth endpoints from cache
    if (isAuthEndpoint(url)) {
      return new Response(...);
    }
    ```

    Add:
    ```javascript
    // Never serve REST API endpoints from cache (prevents stale subscription state)
    if (isRestApiEndpoint(url)) {
      return new Response(
        JSON.stringify({
          error: "Offline",
          message: "API data requires an active network connection",
        }),
        {
          status: 503,
          statusText: "Service Unavailable",
          headers: { "Content-Type": "application/json" },
        }
      );
    }
    ```

    **NOTE on CONTEXT.md offline behavior decision:** The CONTEXT.md says "Show last known subscription state — if user was premium before going offline, keep showing premium content." This is handled at the React layer (SubscriptionContext in Phase 14 will cache the last known premium status in component state/React Query). The service worker's job is to NOT cache REST API responses — the React layer handles the graceful offline experience. The SW offline fallback for REST endpoints returns 503, and React Query's offline behavior will use its in-memory cached value.

    **Do NOT change** the RUNTIME_CACHE_PATTERNS array — the supabase.co regex stays. The exclusion is done at the shouldCache/fallback level, consistent with how auth exclusion works.
