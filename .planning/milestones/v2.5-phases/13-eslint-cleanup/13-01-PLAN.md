```javascript
{
  files: ["**/*.test.{js,jsx}", "**/*.spec.{js,jsx}", "src/test/**/*.{js,jsx}"],
  languageOptions: {
    globals: {
      ...globals.vitest,
    },
  },
},
```
This import `globals` is already available (imported at line 2). The `globals.vitest` object includes: `vi`, `expect`, `it`, `test`, `describe`, `beforeEach`, `afterEach`, `beforeAll`, `afterAll`, `suite`, `bench`, and others. This eliminates ~320 `no-undef` warnings from test files.

**Override 2 — Node globals for config files:**
```javascript
{
  files: ["*.config.{js,cjs}", "*.config.{ts,mts}"],
  languageOptions: {
    globals: {
      ...globals.node,
    },
  },
},
```
This eliminates `no-undef` for `module` in `tailwind.config.js` (1 warning) and `process` in `vite.config.js` (4 warnings).

**Override 3 — `process.env` in src files:**
The files `src/components/games/sight-reading-game/components/VexFlowStaffDisplay.jsx`, `src/contexts/SubscriptionContext.jsx`, `src/services/accountDeletionService.js`, and `src/services/consentService.js` use `process.env.NODE_ENV` which Vite replaces at build time. Add a targeted override:
```javascript
{
  files: [
    "src/components/games/sight-reading-game/components/VexFlowStaffDisplay.jsx",
    "src/contexts/SubscriptionContext.jsx",
    "src/services/accountDeletionService.js",
    "src/services/consentService.js",
  ],
```

**Update existing rules — `no-unused-vars` pattern config:**
In the main config block's `rules` section, change the `"no-unused-vars": "warn"` rule to:
```javascript
"no-unused-vars": ["warn", {
  argsIgnorePattern: "^_",
  varsIgnorePattern: "^_",
  caughtErrorsIgnorePattern: "^_",
}],
```
This tells ESLint to ignore variables prefixed with `_`, enabling Plan 02 to use `_error`, `_unused` patterns for intentionally unused parameters.

**Also update `public/sw.js` handling:**
The service worker file `public/sw.js` uses `self`, `caches`, `clients`, `Response`, `URL`, `fetch` — all ServiceWorker globals. Add an override:
```javascript
{

  languageOptions: {
    globals: {
      ...globals.serviceworker,
    },
  },
},
```

  <action>
For each of the 12 files listed, find the export line that triggers the `react-refresh/only-export-components` warning and add a line-level suppression comment immediately above it.

The warning fires on non-component exports (hooks, constants, reducers) in files that also export components. The suppression format per user decision is:

```javascript
// eslint-disable-next-line react-refresh/only-export-components -- <reason>
```

Use these specific justifications by file type:

**Context files** (AccessibilityContext, AudioContextProvider, ModalContext, SessionTimeoutContext, SettingsContext, SightReadingSessionContext, SubscriptionContext):
These files co-export the Context provider component AND a `useXxx` consumer hook. This is the standard React Context pattern — splitting into separate files would break the encapsulation with no functional benefit. HMR works correctly for these modules.

Justification text: `context provider and hook are co-located by design; splitting would break encapsulation with no HMR benefit`

**Reducer file** (rhythmReducer.jsx):
Exports both the reducer function and action constants. Standard Redux pattern.

Justification text: `reducer and action constants are co-located by Redux convention; HMR-only concern`

**Component files** (NotesRecognitionGame, AnimatedAvatar, Navigation, Toast):
Each of these exports a main component plus helper constants or sub-components. Run `npx eslint <file> 2>&1` on each to identify the exact export line triggering the warning, then add the suppression on that line.

Justification text: `component and helper exports are co-located intentionally; HMR-only dev concern`

IMPORTANT: Only add ONE `eslint-disable-next-line` per warning instance. Each file may have 1 or 2 warnings — check `npx eslint <file>` output to confirm exact lines before adding comments. Do NOT add block-level `/* eslint-disable */`.
